# 易聊社区加载体验优化计划

## 目标

在不改变现有功能的前提下，解决“数据加载慢/体感慢”的问题：

- 首屏、切页、详情、搜索、登录/注册、发帖、评论、后台操作都有明确的加载/发送反馈。
- 加载态参考 linux.do 的体验：不是空白，而是柔和 skeleton、进度感、按钮 busy、局部刷新。
- 降低真实资源加载耗时，尤其是第三方背景图和远程头像。
- 保持当前 UI 风格，不做大重构，不减少功能。
- 修复浏览器返回/站内返回时只显示背景、需要手动刷新的问题。
- 增加“频道”模块：类似 Telegram 频道，但适配当前论坛，频道内容像帖子，普通用户只能浏览和评论，发帖来源只能是管理员手动或接口同步。

## 当前体验观察

测试入口：`http://yhdet.ccocc.cyou/`

### 结论

后端 API 本身并不慢，慢感主要来自前端资源和反馈不足。

- 本地 API：
  - `/api/home` 约 12ms
  - `/api/chrome` 约 12ms
  - `/api/posts/19` 约 11ms
- 公网 API：
  - `/api/home` 约 23ms
  - `/api/chrome` 约 14ms
  - `/api/posts/19` 约 17ms
- 浏览器首屏：
  - `responseEnd` 约 408ms
  - `DOMContentLoaded` 约 578ms
  - `load` 约 3294ms
- 最大拖慢项：
  - `https://t.alcy.cc/fj` 背景图约 2768ms
  - 多个远程头像 400ms～1100ms
  - Font Awesome CDN 约 82ms，可接受但仍是外部依赖

### 主要问题

1. **首页接口快，但视觉上像等待完整数据**
   - `Home()` 在 `!data` 时直接返回 `<Loading />`，页面缺少稳定骨架。
   - 统计、公告、搜索、帖子卡片、侧边栏应该先占位，再替换内容。

2. **第三方背景图拖慢整体 load**
   - `body` 直接使用 `https://t.alcy.cc/fj` 作为 fixed 背景。
   - 这个资源耗时最长，而且失败/慢时影响整体完成感。

3. **远程头像太多，且来源不稳定**
   - 帖子列表一次渲染大量远程头像。
   - 有 `yhdet.top`、`imgs.ovh`、`picui.cn`、`baidu.com/robots.txt` 等来源。
   - 即使 API 快，图片加载慢也会让用户感觉页面慢。

4. **SPA 内部导航到 `/login` 曾出现空白页瞬间/异常**
   - 直达 `/login` 正常。
   - 从帖子详情点击 nav 登录后，snapshot 一度为空白。
   - 需要给路由切换加过渡态，并检查 `AuthPage` 是否受滚动/事件/渲染时机影响。

5. **发送类操作反馈不统一**
   - 登录/注册已有 `提交中...`。
   - 发送邮箱验证码已有 `发送中...` 和倒计时。
   - 发帖、评论没有 loading/disabled 防重复提交。
   - 搜索没有 loading 状态。
   - 后台操作已有 busy，但缺少统一 toast/局部刷新反馈。

6. **错误和成功反馈不够“产品化”**
   - 现在多是卡片内 alert 或文字变化。
   - 应增加轻量 toast：成功、失败、发送中、已保存。

7. **返回时只显示背景，需要刷新才恢复**
   - 现象：站内返回/浏览器返回后，页面主体没有恢复，只剩背景。
   - 高概率原因：SPA 路由状态、异步请求、页面 loading 状态或 `popstate` 同步逻辑在返回时没有正确触发/恢复。
   - 需要在 `useRoute()`、`navigate()`、页面级 `useEffect`、`Loading` 返回路径上做专项修复。
   - 修复目标：浏览器 back/forward、页面内“返回首页”、详情页返回列表都不需要刷新。

8. **缺少频道模块**
   - 需要新增“频道”入口，位置在首页导航后面：`首页 / 频道 / 小游戏 / 音乐 ...`。
   - 频道列表页没有频道时显示“暂无频道”。
   - 频道内容类似帖子流，但发布权限更严格：普通用户不能在频道发帖，只能看帖和评论。
   - 后台需要可以创建/编辑/关闭频道，并支持两种内容模式：接口对接同步、管理员手动发帖。

## 方案总览

按“先体感、再性能、再验证”的顺序做。

### A. 统一加载体验：Skeleton + 页面过渡

新增组件：

- `LoadingOverlay`：顶部细进度条 + 柔和 shimmer，用于路由切换。
- `HomeSkeleton`：首页骨架。
- `PostDetailSkeleton`：详情页骨架。
- `UserSkeleton`：用户页骨架。
- `CardSkeleton` / `LineSkeleton` / `AvatarSkeleton`：通用骨架。
- `ButtonSpinner`：按钮内小 spinner。
- `ToastProvider` 或简单 `useToast()`：全局提示。

linux.do 风格参考点：

- 页面不是黑屏/空白。
- 列表区域先显示灰色块和头像圆形占位。
- 顶部有很细的 loading bar。
- 提交按钮变为 loading，禁用重复提交。
- 成功后有轻提示，不强行弹窗。

### B. 真实性能优化

1. 背景图改成本地/渐进加载
   - 当前：`body { background: #1a1a2e url('https://t.alcy.cc/fj') ... }`
   - 计划：
     - 默认使用本地纯色/渐变背景，立即可见。
     - 如保留随机背景图，改成异步加载：图片加载成功后才加 `.has-bg`。
     - 或直接取消第三方背景，使用轻量 CSS 渐变，最稳。

2. 头像策略
   - 所有头像保留 `loading="lazy" decoding="async"`。
   - 为头像容器加固定尺寸和背景色，避免布局跳动。
   - 首屏列表头像可使用本地 fallback 或 initials/hash 色块，远程头像慢加载。
   - 对明显不适合作头像的 URL（例如 `baidu.com/robots.txt`）后端或前端回退。

3. CDN 依赖
   - Font Awesome 可以暂时保留。
   - 后续可考虑把常用图标替换为 Unicode/SVG，减少外部 CSS 依赖。

4. API 层
   - 当前 API 快，不优先改数据库。
   - 可以给 `/api/home` 加分页或限制最新帖子数量，避免数据变多后变慢。
   - 首页只返回列表必要字段，详情页再拉完整内容。

### C. 交互发送反馈

逐项补齐：

- 搜索：
  - 提交后按钮显示“搜索中...”，禁用。
  - 列表区域显示局部 skeleton，而不是整页闪动。
  - 搜索结果为空给清晰空状态。

- 登录/注册：
  - 保留当前 loading，增加 spinner 图标。
  - 成功后 toast：“登录成功，正在进入首页”。
  - 错误时按钮恢复，错误提示不挤压布局。

- 发送邮箱验证码：
  - 保留倒计时。
  - 增加发送中 spinner 和 toast。

- 发帖：
  - 新增 `posting` 状态。
  - 按钮显示“发布中...”，禁用。
  - 成功后 toast，再跳转详情。
  - 防重复提交。

- 评论：
  - 新增 `commenting` 状态。
  - 按钮显示“发表中...”，禁用。
  - 成功后先本地插入或局部刷新，避免用户感觉无响应。

- 后台：
  - busy 时对应按钮禁用。
  - 操作成功/失败 toast。
  - 切 tab 时显示 table/list skeleton。

### D. 返回/后退空白修复

专项修复目标：任何返回动作都不能只剩背景。

计划检查和修复点：

1. `useRoute()`
   - `popstate` 和自定义 `app:navigate` 都必须稳定触发 `setPath(currentRoute())`。
   - 不要让 `requestAnimationFrame` 被取消后丢失更新。
   - `window.scrollTo({ behavior: 'instant' })` 兼容性可能有问题，改成安全写法：`window.scrollTo(0, 0)` 或 try/catch。

2. `App` 路由渲染
   - 所有路由必须有兜底返回内容，不能在异常状态下返回空 fragment。
   - 页面组件出错时显示错误卡片，而不是空白。

3. 页面数据加载
   - `Home`、`PostDetail`、`UserPage`、`SimpleSection` 的 `useEffect` 要处理卸载和路由切换。
   - 避免旧请求完成后覆盖新页面状态。
   - 加 `cancelled` 标记或 `AbortController`。

4. 视觉兜底
   - 路由切换期间显示 skeleton/进度条。
   - 即使数据请求失败，也显示导航、页面框架、错误提示。

验证动作：

- 首页 → 帖子详情 → 浏览器返回。
- 首页 → 登录 → 浏览器返回。
- 详情 → 登录 → 浏览器返回。
- 首页 → 用户页 → 返回首页。
- 快速连续点击导航和返回。

### E. 新增频道模块

频道定位：类似 Telegram 频道，但融入当前论坛样式。频道消息/文章看起来像帖子，用户只能阅读和评论，不能在频道主动发帖。

#### 前台信息架构

导航：

- 首页
- 频道
- 小游戏
- 音乐
- 后台（管理员）

页面：

- `/channels`：频道列表
  - 有频道：卡片列表，显示频道名、简介、模式、最近更新时间、文章数。
  - 无频道：显示空状态“暂无频道”。
- `/channels/:id`：频道详情/频道帖子流
  - 顶部频道信息卡：名称、简介、模式标签、同步状态。
  - 内容列表：每条频道内容使用帖子卡片样式。
  - 普通用户无“发帖”按钮。
  - 管理员在手动模式频道看到“发布频道内容”按钮。
- `/channels/:channelId/posts/:postId`：频道内容详情
  - 内容样式接近帖子详情。
  - 评论区复用当前帖子评论体验。
  - 普通用户可以评论；是否允许游客评论保持当前站点规则：需要登录则沿用登录规则。

#### 频道模式

1. 管理员手动发帖模式：`manual`
   - 后台创建频道后，管理员可以在频道内发布内容。
   - 发布表单类似发帖，但作者展示为管理员/频道名。
   - 普通用户只能阅读和评论。

2. 接口对接模式：`api`
   - 用于对接外部来源，例如 TG 管理后台里的 linux.do 频道。
   - 后台配置接口参数：
     - `source_type`：例如 `tg_monitor_linuxdo`、`custom_api`
     - `endpoint_url`：接口地址
     - `auth_type`：none / bearer / header
     - `auth_secret`：密文/环境变量引用，后台显示掩码
     - `poll_interval`：同步间隔，先可手动触发，定时同步后续再加
     - `mapping`：标题、内容、作者、时间、外链字段映射
   - 第一版建议先做“后台手动同步按钮”，不马上做复杂定时任务。

#### 后台管理

在后台增加 `频道` 菜单，不替代现有菜单。

功能：

- 创建频道
  - 名称
  - slug
  - 简介
  - 模式：管理员手动 / 接口对接
  - 是否启用
- 编辑频道
- 删除/停用频道
- 查看频道内容列表
- 手动发布频道内容（manual）
- 接口模式配置和“测试连接/立即同步”
- 同步日志：成功数、失败原因、最后同步时间

#### 数据模型建议

SQLite 新增表：

```sql
CREATE TABLE channels (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  slug TEXT UNIQUE NOT NULL,
  description TEXT DEFAULT '',
  mode TEXT NOT NULL DEFAULT 'manual',
  enabled INTEGER NOT NULL DEFAULT 1,
  source_type TEXT DEFAULT '',
  endpoint_url TEXT DEFAULT '',
  auth_type TEXT DEFAULT 'none',
  auth_secret_ref TEXT DEFAULT '',
  mapping_json TEXT DEFAULT '{}',
  last_sync_at TEXT DEFAULT '',
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE channel_posts (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  channel_id INTEGER NOT NULL,
  title TEXT NOT NULL,
  content TEXT NOT NULL,
  author_name TEXT DEFAULT '管理员',
  external_id TEXT DEFAULT '',
  external_url TEXT DEFAULT '',
  source_payload_json TEXT DEFAULT '{}',
  views INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  UNIQUE(channel_id, external_id),
  FOREIGN KEY(channel_id) REFERENCES channels(id)
);

CREATE TABLE channel_comments (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  channel_post_id INTEGER NOT NULL,
  user_id INTEGER NOT NULL,
  content TEXT NOT NULL,
  created_at TEXT NOT NULL,
  FOREIGN KEY(channel_post_id) REFERENCES channel_posts(id),
  FOREIGN KEY(user_id) REFERENCES users(id)
);
```

可选：如果想最大复用当前帖子逻辑，也可以给现有 `posts` 增加 `channel_id`、`post_type` 字段。但为了避免影响普通论坛帖子，第一版推荐独立 `channel_posts`。

#### API 设计

前台：

- `GET /api/channels`
- `GET /api/channels/{id_or_slug}`
- `GET /api/channels/{id_or_slug}/posts`
- `GET /api/channel_posts/{post_id}`
- `POST /api/channel_posts/{post_id}/comments`

后台：

- `GET /api/admin/channels`
- `POST /api/admin/channels`
- `PUT /api/admin/channels/{id}`
- `DELETE /api/admin/channels/{id}` 或 `POST /api/admin/channels/{id}/disable`
- `POST /api/admin/channels/{id}/posts`（manual 模式）
- `POST /api/admin/channels/{id}/test-source`（api 模式）
- `POST /api/admin/channels/{id}/sync`（api 模式）

#### 权限规则

- 普通用户：
  - 可查看频道列表、频道内容、频道详情。
  - 可评论频道内容（沿用登录要求）。
  - 不可发布频道内容。
- 管理员：
  - 可创建/编辑/删除频道。
  - 可手动发布频道内容。
  - 可配置接口、测试连接、同步内容。
- 接口同步：
  - 只能写入 `channel_posts`。
  - 不影响普通帖子列表，除非后续明确需要把频道内容混入首页。

#### 首页关系

- 首页后面增加“频道”导航入口。
- 首页本身先不混入频道内容，避免信息流混乱。
- 可在首页侧栏增加“推荐频道/最新频道内容”，但作为第二阶段。

## 分阶段执行计划

## 风险和取舍

1. **不要为了动画牺牲速度**
   - skeleton 和 spinner 要轻量。
   - 不做复杂 Lottie 或大型动画库。

2. **不要全局遮罩阻塞用户**
   - 首页/列表用局部 skeleton。
   - 发送操作只禁用相关按钮，不锁死整个页面。

3. **保留现有功能完整性**
   - 管理后台菜单、登录注册、验证码、发帖评论、搜索都不能少。

4. **背景图取舍**
   - 如果用户想保留随机图，可以异步加载。
   - 如果优先速度，建议直接用 CSS 渐变背景。

## 推荐执行顺序

推荐分两次执行，降低风险：

第一批先解决体验问题：

1. 骨架屏/按钮 loading/toast。
2. 移除或延迟第三方背景图。
3. 头像占位和 fallback。
4. 修复返回只剩背景的问题。
5. 构建、重启、浏览器验证。

第二批再做频道模块：

1. 数据库表和频道 API。
2. 前台频道列表/详情/频道内容详情。
3. 后台频道管理和手动发布。
4. 接口对接配置、测试连接、手动同步。
5. 权限和回归验证。

这样先快速修复当前用户能感知的卡顿/空白，再做较大的频道功能，避免一次改太大导致现有功能回归。
